Project 03 — Cloud Infrastructure

Virtual Active Directory & Site-to-Site VPN on Azure

Microsoft Azure · Windows Server 2022 · Active Directory · Domain Controllers · GPO · Azure VPN Gateway · Entra ID · Hybrid Cloud
Microsoft Azure Windows Server 2022 Active Directory Domain Controllers GPO Azure VPN Gateway Entra ID Hybrid Cloud
What this proves — 30 second summary

This project demonstrates the ability to deploy and manage cloud infrastructure on Microsoft Azure — specifically virtualised Active Directory and a site-to-site VPN, which are foundational components of any hybrid enterprise environment. The skill set here is what separates a pure L1 support technician from someone capable of junior sysadmin or cloud support work.

Project Overview

Many companies run a hybrid environment — on-premises Active Directory synced to Azure/Entra ID in the cloud. This project simulates that by deploying a full AD infrastructure on Azure VMs (Windows Server 2022), then connecting it securely to a cloud-only Entra ID tenant via a site-to-site VPN.

The project covers cloud infrastructure deployment (Azure VMs, VNets, resource groups), Active Directory setup (domain controllers, OUs, GPOs), and secure connectivity (Azure VPN Gateway, local network gateway, IPSec tunnel).

Step-by-Step Breakdown
1
Set Up Azure Resource Group and Virtual Network
›

Created the foundational Azure infrastructure before deploying any VMs:

  • Created a Resource Group to contain all project resources (easier to manage and delete as a unit)
  • Deployed a Virtual Network (VNet) with address space 10.0.0.0/16
  • Created subnets: 10.0.1.0/24 for domain controllers, 10.0.2.0/24 for member servers, and a GatewaySubnet (required for VPN gateway)
  • Configured Network Security Groups (NSGs) to allow RDP (port 3389) from a specific IP only, and block all other inbound traffic
2
Deploy Windows Server 2022 VMs
›

Deployed two Windows Server 2022 VMs to simulate a primary and secondary domain controller:

  • Selected Windows Server 2022 Datacenter from the Azure Marketplace
  • Chose B2s VM size (2 vCPUs, 4GB RAM) — sufficient for a lab domain controller
  • Assigned static private IPs to both VMs — DCs must not use DHCP as other machines use their IP as the DNS server
  • Disabled public IPs post-setup and accessed via Azure Bastion for security
  • Ran Windows Update and applied all patches before promoting to DC
3
Promote to Domain Controller — Active Directory Setup
›

Installed the AD DS role and promoted both servers to domain controllers:

  • Installed Active Directory Domain Services (AD DS) role via Server Manager
  • Promoted the first VM to a DC, creating a new forest and domain: aayushlab.local
  • Promoted the second VM as an additional DC in the same domain — provides redundancy
  • Configured DNS on both DCs — pointed all VMs to use the primary DC's private IP as DNS server
  • Verified AD replication between both DCs using repadmin /showrepl
4
Build OU Structure and Configure GPOs
›

Created a realistic organisational structure in Active Directory and applied Group Policy to control it:

  • Built OU hierarchy: aayushlab.local → Departments → IT, Finance, HR, Management
  • Created user accounts and computer accounts in the correct OUs
  • Built and linked GPOs at the OU level:
    • Password Policy GPO: 12-char minimum, complexity required, 90-day expiry
    • Desktop Lockdown GPO: removed Control Panel, restricted USB storage
    • Drive Mapping GPO: auto-mapped network drives based on department membership
    • Security GPO: disabled legacy SMBv1, enabled audit logging for logon events
  • Ran gpupdate /force on test machines and verified policies applied with gpresult /r
5
Deploy Azure VPN Gateway
›

Set up the Azure-side of the site-to-site VPN to connect the on-premises (simulated) network to the Azure VNet:

  • Created an Azure VPN Gateway in the GatewaySubnet (deployment takes ~30 minutes)
  • Selected Route-based VPN type (required for Entra ID integration and modern IKEv2)
  • Created a Local Network Gateway representing the on-premises side — specified the on-prem public IP and address space
  • Created the VPN Connection between the two gateways using a shared key (pre-shared key)
  • Configured BGP for dynamic routing (optional but more realistic for enterprise simulation)
6
Integrate with Microsoft Entra ID (Hybrid Identity)
›

Configured hybrid identity so the on-premises AD users sync to the Entra ID (cloud) tenant:

  • Installed Microsoft Entra Connect on a member server in the Azure VNet
  • Ran the Entra Connect wizard — selected Password Hash Synchronisation as the sign-in method
  • Configured the OUs to sync (excluded service accounts and test accounts from sync scope)
  • Verified users appeared in Entra ID admin centre with synced from on-premises badge
  • Tested hybrid sign-in: a user created in on-prem AD could sign into Microsoft 365 with the same credentials
  • Confirmed Conditional Access policies from the M365 project applied to synced users correctly
