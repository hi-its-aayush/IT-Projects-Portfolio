What this proves — 30 second summary

This project demonstrates practical networking knowledge — specifically how enterprise networks are segmented to separate different types of traffic (user data, VoIP, guest Wi-Fi, management). Network segmentation is a core concept in IT support because it affects how users connect, why some devices can't reach others, and how VoIP quality is maintained. Understanding VLANs is expected at L2 and above.

Project Overview

Designed a multi-VLAN enterprise network topology in Cisco Packet Tracer, simulating the type of network architecture found in a mid-sized business with multiple departments. The goal was to segment traffic correctly, enable controlled inter-VLAN communication via a Layer 3 switch, and validate that each segment worked as intended.

This directly mirrors the Cisco Meraki multi-site network infrastructure I manage professionally — understanding the logical design behind what a tool like Meraki abstracts away is what separates an operator from someone who actually understands what's happening.

Step-by-Step Breakdown
1
Plan the Topology and IP Addressing Scheme
›

Before touching Packet Tracer, mapped out the full network design on paper:

  • Identified how many VLANs were needed and what traffic each would carry
  • Assigned a /24 subnet to each VLAN to keep addressing simple and scalable
  • Planned that all inter-VLAN routing would happen at the Layer 3 switch, not the router (router-on-a-stick was tested but L3 switch is more scalable)
  • Decided Guest VLAN would have no route to any internal VLAN — internet-only via ACL
2
Build the Physical Topology in Packet Tracer
›

Constructed the network in Cisco Packet Tracer using:

  • 1x Layer 3 switch (Cisco 3560) as the core — handles inter-VLAN routing
  • 4x Layer 2 access switches — one per floor/zone, connected to core via trunk links
  • 1x Router — for upstream internet connectivity
  • End devices: PCs, IP phones, laptops, and a server in each relevant VLAN
  • Connected each access switch to the L3 core switch using GigabitEthernet trunk ports
3
Configure VLANs and Assign Ports on All Switches
›

Created VLANs on all switches and assigned ports as either access or trunk:

  • Created VLANs 10, 20, 30, 40 in the VLAN database on every switch (vlan 10, name DATA)
  • Set access ports for end devices: switchport mode access, switchport access vlan 10
  • Set trunk ports between switches: switchport mode trunk, allowed all VLANs across the uplinks
  • Configured voice VLAN on ports connecting to IP phones: switchport voice vlan 20 alongside the data VLAN — allows a single port to carry both PC and phone traffic
  • Verified VLAN assignments with show vlan brief and show interfaces trunk
4
Enable Inter-VLAN Routing on the Layer 3 Switch
›

Configured the L3 switch as the default gateway for all internal VLANs:

  • Created Switched Virtual Interfaces (SVIs) for each VLAN — these act as the default gateway for devices in that VLAN
  • Example: interface vlan 10 → ip address 192.168.10.1 255.255.255.0 → no shutdown
  • Enabled IP routing on the L3 switch: ip routing
  • Configured a default route pointing to the router for internet traffic: ip route 0.0.0.0 0.0.0.0 [router IP]
  • Verified routing table with show ip route
5
Configure DHCP Per VLAN
›

Set up DHCP pools on the Layer 3 switch so each VLAN gets IP addresses automatically:

  • Created a DHCP pool for each VLAN with the correct network, default gateway, and DNS server
  • Excluded the gateway IP and first 10 addresses from each pool (reserved for static assignment)
  • Example for VLAN 10: pool name DATA, network 192.168.10.0/24, default-router 192.168.10.1, dns-server 8.8.8.8
  • Verified a PC in VLAN 10 received a DHCP address: ipconfig /all in the PC terminal
6
Apply ACL to Isolate Guest VLAN
›

The Guest VLAN should have internet access only — it must not be able to reach any internal resources:

  • Created an Access Control List (ACL) on the L3 switch to deny traffic from VLAN 40 to any internal subnet (10.x, 192.168.10-30.x)
  • Permitted traffic from VLAN 40 to any other destination (internet via the router)
  • Applied the ACL inbound on the VLAN 40 SVI: ip access-group GUEST_ACL in
  • Tested by pinging from a Guest PC to a Data VLAN PC — confirmed denied. Pinged 8.8.8.8 — confirmed permitted.
7
End-to-End Connectivity Testing
›

Validated the full design using Packet Tracer's simulation mode and the PDU (packet) tool:

  • VLAN 10 PC → VLAN 30 Server: permitted (users can access the file server)
  • VLAN 40 Guest → VLAN 10 PC: denied by ACL (guest isolation working)
  • VLAN 20 IP Phone → VLAN 10 PC: permitted (voice-to-data allowed)
  • All VLANs → Router (internet): permitted
  • Used show ip arp and ping across all VLANs to verify routing table and connectivity
